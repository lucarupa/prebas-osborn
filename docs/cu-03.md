### CU-03: Aprobar o rechazar una versión

**Actor principal:** Revisor (`reviewer`) o Cliente (`client`)  
**Precondición:** Existe una versión en estado `pending_review` que aún no tiene decisión final.  
**Flujo principal:**
1. El revisor/cliente envía la decisión (`approved` o `rejected`) con un comentario opcional.
2. El sistema crea el registro en `approvals`.
3. Si la decisión es `approved`, el asset pasa a estado `approved`.
4. Si es `rejected`, el asset permanece/vuelve a `pending_review` para que el diseñador suba una nueva versión.
5. Se notifica al diseñador.

**Postcondición:** El estado del asset refleja la última decisión.
